# Track 3 Alignment Doctrine

This doctrine freezes the submission frame after the v3 packaging pass. The project should be presented as a verification layer for AI-written software, not as a generic coding benchmark and not as a broad formal-methods revival.

## North Star

The submission is an Inspect-native vericoding prototype in which specifications act as hidden executable ranking and rejection oracles for autonomous code acceptance on narrow-waist security-critical components.

The unit of analysis is a candidate or selected final artifact evaluated under visible checks, hidden executable checks, and review-boundary rules, resulting in `auto_accept`, `auto_reject`, or `escalate_to_review`.

## Primary Denominator

The primary paper object remains the 24-task owned narrow-waist suite:

- 8 dev tasks
- 16 confirmatory tasks
- all headline claim thresholds anchored here

The 24-task expansion suite broadens mechanism and review-boundary coverage. It does not redefine the primary denominator or retune thresholds.

## Claim Order

The judge-facing claim order is:

1. Visible tests are insufficient for autonomous acceptance.
2. Hidden executable evaluators materially change accept/reject decisions.
3. Secure rejection is the cleanest positive trust-boundary surface.
4. Ranking helps only when the candidate bank contains hidden-correct support.
5. Some executable-pass artifacts still require explicit human review escalation.
6. Repair is a null or bounded secondary result unless it beats an equal-cost fresh-generation baseline.

## Secondary Surfaces

The following surfaces are contextual unless they sharpen the same trust-boundary object:

- v2 oracle/hybrid lineage
- Sprint 9 SCBench lineage
- Sprint 10 Terminal-Bench lineage
- bounded formal overlay
- Dafny history

None of these surfaces replaces the v3 primary denominator.

## Formal Extension Rule

Formal work may appear only as a bounded sidecar. It must strengthen a trust-boundary or review-boundary claim and must not reopen the denominator, redefine the paper, or start a broad Dafny generation campaign.

## Forbidden Reframes

Future submission writing should not present the project as mainly:

- SlopBench
- cyclomatic-complexity reduction
- generic benchmark comparison
- broad formal synthesis
- a proof-first Dafny system

Older SlopBench and Dafny work may remain as lineage, but the front-door object is executable trust-boundary triage.
